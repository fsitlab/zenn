# 23 比喩の検証 — ステートレスの上に何を建てるか

## 0. 位置づけと中心命題

21（説明資料）・22（モックUI）を実際に作って比喩を検証した結果と、外部調査（onboarding／USB-C／指揮者／ロードアウト／会社経営）を統合する。本書の背骨は、検証の過程で確定した次の命題である：

> **LLMは本質的にステートレスである。人の比喩は、その上にステートフルな実体（社員・人格・同一性）を仮構しようとする。カードゲームの比喩は、ステートレスでエフェメラルなまま——カードの組み合わせで効果を発揮し、デッキ編集でユースケースをカスタムする。**

良い比喩の基準は一貫して「そこからアーキと解決策が見えるか（生成力）」。本書はこの基準に「**基盤の物理に適合しているか（同型性）**」を加える。生成力は同型性の帰結だからである——基盤と同じ形の比喩だけが、嘘のない操作を生成できる。

---

## 1. 本質の特定 — スキル・MCP・ハーネスとは何か

### 1.1 一次原理：LLM＝ステートレス関数

推論1回の実態は `f(コンテキスト窓) → 出力` である。呼び出し間に記憶はなく、「会話」も「エージェント」も「セッション」も、**毎回、窓の外に置かれた状態を窓の中へ再シリアライズして見せている**だけだ。状態は常に外にある——ファイルに、DBに、lockに、そして次のプロンプトに。これがすべての出発点であり、純正メモリもサブエージェントもRAGも、この物理の上の工夫にすぎない。

### 1.2 本質は「コンテキストウィンドウゲーム」、ツール活用はその一手

「タスク処理のツール活用」と「コンテキストウィンドウゲーム」のどちらが本質か——**後者が深層、前者は表層**である。関係はこうだ：

| 観点 | コンテキストウィンドウゲーム | タスク処理のツール活用 |
|---|---|---|
| 問うていること | 毎回の呼び出しで、有限の窓に**何を入れるか**（選択・圧縮・段階開示・持ち越し） | 出力を使って**外界に何をするか**（実行・副作用） |
| 位置 | ループの入力面＋状態管理＝設計の主戦場 | ループの出力面＝ゲームの一手 |
| 失敗の形 | context rot、窓あふれ、関係ない情報の混入、状態の蒸発 | ツール誤用、宣言外アクション |
| 統制の対象 | 何が窓に入り得るか（＝構成） | 何が外に出られるか（＝境界） |

ツール呼び出し自体、「ツール定義を窓に入れ、出力をパースし、結果を次の窓に戻す」という**窓ゲームの往復**として実装されている。だから正確に言えば本質はこうなる——**状態を窓の外に資産として置き、毎回の呼び出しで窓を再構成する競技。** ツール活用はその競技のプレイの一種である。

### 1.3 三つの規格の正体（一行ずつ）

- **スキル（SKILL.md）**＝手続き知識を窓へ供給する規格。段階開示（メタデータ常駐→本文は必要時ロード）は、窓ゲームにおける「手札を温存し、必要なターンに引く」技法そのもの
- **MCP**＝外界と窓のI/O標準。世界の状態を窓に持ち込み、行動を外に出す共通ポート
- **ハーネス**＝窓の周回ループ。何を入れ（補給）、出力をどう実行し、何を次の窓へ持ち越すか。16・18で示した通り、補給側はモデル進歩に食われ続け、**残るのは境界と私有文脈の供給だけ**

### 1.4 「ゲーム」という語は飾りではない

有限資源（窓）、毎ターンの選択（何を引き、何を伏せるか）、ターン終了でのリセット、構成（デッキ）と1プレイ（対戦）の分離——**この構造定義は、そのままTCGの構造定義である。** 「コンテキストウィンドウゲーム」という直観は、比喩を探した結果ではなく、基盤の物理を言い当てた結果としてカードゲームに着地する。

## 2. ステートレス・テーゼ — 二つの比喩の物理適合

### 2.1 カードゲームは基盤と同型である

| 基盤の物理（LLM運用） | TCGの構造 |
|---|---|
| モデルはステートレス。テキスト（重み）は呼び出しで変化しない | カードテキストは不変。ゲーム中にカードが書き換わることはない |
| 状態は実行スコープ限り。終われば破棄 | 盤面・ライフ・手札は1戦限り。終わればリセット |
| 毎呼び出しで窓を再構成 | 毎ゲームのシャッフルとドロー |
| 窓は有限 | マナ・デッキ枚数・手札上限 |
| 段階開示（必要時にロード） | 伏せ札・ドローして初めて公開 |
| 能力は合成で発現（単体プロンプトは無力でも組合せで動く） | コンボ：1枚では紙、場の組合せで効果 |
| 永続するのは構成（recipe/lock）だけ | 永続するのはデッキリストだけ。盤面は残らない |
| 改善＝差し替え、複製＝同一構成の並列展開 | 差し替えと、同名カードの複数投入・デッキコピー |
| 成熟知の重みへの蒸留 | キーワード能力化（実績ある効果のルールブック収録） |

**カードに状態がないこと**が決定的だ。だからdigestが貼れ、凍結でき、複製でき、監査できる。比喩の各操作が、工学の正当な操作に1対1で翻訳される。

### 2.2 人の比喩は、操作の段になると嘘をつく

ご指摘の「人の複製？再教育？？」を表にすると、破綻の場所が正確に見える：

| やりたい工学操作 | デッキの語彙 | 人・会社の語彙 | 破綻の仕方 |
|---|---|---|---|
| スケールアウト（同一構成を100並列） | 同じデッキで100卓に着く | 同じ社員を100人**複製** | クローンの不気味さで思考が止まる。TCGでは同名カード複数投入が正常ルール |
| 改善 | 1枚差し替え→再凍結 | **再教育**・研修 | 重みは1bitも変わっていないのに「教育した」と錯覚し、検証（eval）が省かれる |
| ロールバック | 旧lockで再召喚 | 記憶を消す？降格？ | 人に存在しない操作。比喩が沈黙し、設計からロールバックが落ちる |
| 厳密な再現 | digest照合 | 「同じ人にもう一度やらせる」 | 人は二度と同じではない。再現性の要求自体が比喩から消える |
| 退役 | スタン落ち／重みへ収録して名誉退役 | 解雇・退職 | 知識が人と一緒に出ていく設計を生む（US-4：純正メモリと消えた記憶） |
| 監査 | デッキリスト提出・実体照合 | 人物の信用・性善説 | 監査人は「この社員を信じた」を証拠と認めない |

人の比喩の根本問題は、**人がステートフルな存在そのもの**であることだ。記憶・経験・同一性が本体である比喩を、状態を持たない基盤に被せると、状態をモデル側に求める設計——長命エージェント、肥大するメモリ、人格の継続——が生成される。実証は揃っている：長期稼働での同一性の漂流（AnthropicのProject Vendで観察）、context rot、そして役職エージェントの組織図を建てたChatDev/MetaGPT系がモデル進歩に食われ、**残骸として資産化されたのはSOP→SKILL.md＝カードだった**という2023→26年の歴史（16）。

### 2.3 ただし、人の比喩が唯一正しい場所がある

**重みの中**である。事前学習とファインチューニングは本当に「教育」だ——経験が消えない形で主体に刻まれる。だから18の学習レーンは、比喩の言葉で言えば**「カードの世界」から「教育の世界」への橋**であり、learnableカードは「教材」、学習SBOMは「履修証明」と読める。人の比喩は重みのライフサイクル専用であり、運用層に持ち出した瞬間に嘘になる。

## 3. 判定と配置 — 四層モデル

前章までで、比喩は競合ではなく**層の取り違え**が事故の原因だと分かる。正しい配置：

| 層 | 実体 | 合う比喩 | 取り違えると起きる事故 |
|---|---|---|---|
| 重み | 事前学習・FT（変化が主体に刻まれる） | **教育・人**（ここだけ人が正しい） | デッキ比喩を持ち込むと「重みも差し替え可能」と錯覚（ベンダーモデルでは不可能＝NFR-8の明記事項） |
| 資産・構成 | カード／デッキ／recipe・lock | **デッキ構築** | 人比喩→知識が暗黙知のまま人格に滞留し、異動・退職・モデル更新で蒸発 |
| 実行 | エフェメラルな召喚と破棄 | **召喚（対戦1回）** | 人比喩→長命エージェント→context rot・同一性漂流・残骸の汚染 |
| 経営・運用 | 委譲・レビュー・責任・エスカレーション | **会社経営** | デッキ比喩は人間の一日を語れない。「Claude Codeで会社経営」の正当な手柄はここ |

スキルの構成・エージェントの構成・ライフサイクルという問いは、すべて第2層（一部第3層）の問いである。**だから本質はデッキ構築**——ただし第1層の入口（教育＝学習レーン）と第4層の出口（経営＝人間の運用UX）を、それぞれの比喩に接続して使う。

## 4. 検証の記録 — 生成実績・ほころび・競合比喩（要約収録）

**生成実績**（比喩→設計の変換が実際に起きた箇所）：デッキリスト提出→lock凍結／マナ→コンテキスト事前会計／フォーマット→ISMAP閉域のfail-closed検証／禁止改訂の作法→失効貫通＋S18運用／スターター・優勝デッキ→「コピーして1枚差し替え」動線／**統率者→モデル駆動リント**（比喩が設計に先行）／**カードの表裏→枠＝統制・裏＝SKILL.md原文**（22の裏返しUIは物理カードの二面性から即時に導出）／Timmy-Johnny-Spike→ペルソナ分析／キーワード能力化→学習レーンの語彙。さらに「許可データソース一覧カード」という設計ミスは、統率者戦の固有色ルール（入れた色の集合＝使える色）を厳密適用していれば**比喩が事前に防げた**——比喩が検証装置として働いた反証的な証拠。

**ほころび**（正直な限界）：①これは「対戦」ではなく**デッキ構築**の比喩——手札・ターン・勝敗は写像なし、実行はエフェメラル／コンテナの語彙で補う。②レアリティ・ガチャ・トレード価格は能動的に排除が必要（実例：RPGロードアウトUI「World of Claudecraft」はトークンコストでレアリティを導入しており、輸入誘惑の強さを示す）。③学習昇格（キーワード能力化）は造語＝比喩の自然な射程の外。

**競合比喩の調査結果**（要点のみ）：

| 比喩 | 出どころ | 担える層 | 限界 |
|---|---|---|---|
| 新入社員onboarding資料 | Anthropic公式（スキル＝新人向けオンボーディングガイド） | 第2層の**中身**（SKILL.mdの書き味）に限り最良 | 構成・統制・コスト・流通を生成しない |
| USB-C | MCP公式級アナロジー | 接続標準の説明 | 「標準だけでは認証・権限・統治は残る」——その残余がデッキの守備範囲 |
| 指揮者とオーケストラ | オーケストレーション解説の定番 | 協調の直感 | **固定譜面＝ワークフローを生成して腐る**（Microsoft Conductorは構造の定義時固定を明言）。比喩選びは設計選び、の悪い側の実例 |
| RPGロードアウト・編成 | World of Claudecraft等、日本では「パーティ編成」 | 構成・コスト（最有力の親戚） | **統治語彙がゼロ**。デッキリスト提出・ジャッジ・禁止改訂・フォーマットに相当する制度がない |
| 会社経営・社員 | ChatDev/MetaGPT〜「Claude Codeで会社経営」 | 第4層（人間の運用UX）・信頼段階（試用期間＝検疫） | 第2・3層に持ち込むとステートフル仮構（2.2の操作破綻表） |

**テーゼ**：TCGが勝つ理由は遊びの比喩だからではない。**「構成物の規制制度」（デッキリスト・ジャッジ・禁止改訂・フォーマット・エラッタ・ローテーション）を30年運用した唯一のゲームジャンル**だからである。統治が要る製品には、統治語彙を内蔵した比喩が要る。

## 5. 帰結 — コアの再導出

18のコア（3不変量＋1動詞）は、ステートレス・テーゼの系として再導出できる：

- 状態は窓の外に置く → 置かれた資産には同一性が要る → **digest＋凍結（不変量①）**
- 窓に入り得るものの全体＝エージェントの能力 → 構成が権限境界 → **デッキ単位（不変量②）**
- 窓に渡るのは中身そのもの。枠の情報は渡らない → 中身は標準のまま、統制は外付け → **スーパーセット規律（不変量③）**
- ステートレスゆえ「成長」は存在せず、変化はすべて構成の編集 → **swap（動詞）**

つまり「シンプルだから残る」（18）の根はここにある——**コアが小さいのは、基盤の物理（ステートレス）が要求する不変量がもともと少ないから**だ。人の比喩で建てると、存在しない状態を管理する機構（人格・記憶・成長の擬装）が積み上がり、コアが太り、腐る。

一行で締める：**LLMの本質はステートレスな窓ゲームであり、その上に建ててよいのは状態ではなく構成である。構成の比喩としてカードゲームを選ぶのは趣味ではなく、物理への適合である。**
