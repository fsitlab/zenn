## 0. 前提：このメインコンセプト原稿の特徴

- 既存ドラフトは「序章→1戦観戦→**もの／時間／空間**の三次元＋**法則**→上級ルール→人間の席→終章」という、極めて綺麗な骨格を**既に持っている**。

- 性質：**メタファー駆動**。常に「ゲームの言葉」と「AIの実体」の二層が並走する。ここをどう見せるかが構造の肝。

## 1. リサーチ：通常の本によくある構成

### 1-A. 物理的な三部構成（製本の定番）
- **前付け（Front matter）**：タイトル / まえがき・はじめに / 対象読者 / 本書の読み方 / 目次
- **本文（Body）**：部（Part）＞章（Chapter）＞節（Section）
- **後付け（Back matter）**：用語集 / FAQ / 付録 / 参考文献 / あとがき / 索引

### 1-B. 実用書・技術書・ビジネス書でよく使う「論理展開パターン」
1. **問題→解決（Problem-Solution）**：困りごと提示→原因→解法
2. **概念→応用（Concept-First）**：基礎モデルを先に立て、後半で実践
3. **時系列／成長段階（Chronological / Ladder）**：歴史順・レベル順
4. **全体→部分→全体（Whole-Part-Whole）**：全体像→分解→再統合（★メタファー本と相性◎）
5. **既知→未知（Known-to-Unknown）**：読者の馴染みある物事から橋渡し（★本書はまさにこれ）
6. **反復らせん（Spiral）**：同じ概念を浅→深と何周もする
7. **FAQ／逆説つぶし**：よくある誤解を一つずつ壊す

### 1-C. 「導入のつかみ」定番テンプレ
- 失敗あるある / 1シーンの実況 / 衝撃的な一文（テーゼ） / 用語の言い換え対比 / 約束（読み終えて得られるもの）

### 1-D. 章の内部テンプレ（揃えると読みやすい）
- フック（具体シーン）→ ルール提示 → 種明かし（実体との対応）→ 誤解つぶし → まとめ一行

## 2. この本の趣旨的に「必要そうな構造」

- **二層の往復装置**：各セクションで「ゲーム語で語る」→「AI実体に翻訳する」を制度化する仕掛けが要る（例：各章末の「種明かしコラム」、巻末の対応表）。
- **既知→未知（1-B-5）を全体の背骨に**：カードゲームの常識を入口に、AIの非常識を理解させる。
- **全体→部分→全体（1-B-4）**：序章で1戦の全体像→もの/時間/空間で分解→終章で再統合（既存ドラフトが既にこの形）。
- **読者の二極化対応**：①カードゲーム知ってるがAI知らない／②AI触るがカードゲーム知らない／③どちらも素人。→「最小ルール説明」セクションの要否を判断。
- **誤解つぶしを縦糸に**：育つAI／覚えてるAI／考えといて／自律AI幻想…が全編の通奏低音。独立コラム化の候補。
- **実践への着地が薄い**：既存ドラフトは思想・モデル説明が中心。「で、明日どう使う？」の How-to を足すか否かが最大の分岐。
- **誠実さの担保**：「比喩はどこまで正しく、どこから崩れるか」を書くセクションがあると信頼が増す。

> カードゲームに固有語（マナ・墓地・トークン…）があるのと同じ温度で、
> **AIの専門用語を"ゲームの用語"として正しく定義・解説する**。
> 初心者はゲームとして腑に落ち、専門家が読んでも定義が正確で誤りがない。
> ＝二層（比喩×実体）を曖昧にせず、用語レベルで一対一に対応させる。

- Q1 各ゲーム語に「正式なAI用語」と「正確な一文定義」を必ず併記する様式
- Q2 専門用語を「初出で太字・定義 → 以後ゲーム語で運用」する導入ルール
- Q3 比喩が崩れる箇所は正直に注記（例：トークンの二義）
- Q4 章末「正確に言うと」ボックス（専門家向けの厳密版）
- Q5 巻末・正式用語索引（AI用語→本書のゲーム語が引ける逆引き）

#### 用語対応表ドラフト（ゲーム語 ⇄ AI正式用語 ⇄ 正確な定義）
> ※定義は専門家チェック前提のたたき台。要・事実確認。

| ゲーム語 | AI正式用語 | 正確な定義（一文） |
|---|---|---|
| 打ち手／AIカード | LLM（大規模言語モデル）／モデル本体 | 学習済みパラメータ（重み）で次トークンを予測する関数。推論時に重みは更新されない。 |
| 召喚 | 推論の開始／セッション・実行の起動 | プロンプトをモデルに渡し、生成を始めること。 |
| 盤面 | コンテキストウィンドウ | モデルが一度に参照できる入力トークンの有限な領域。ここに無い情報は参照されない。 |
| カード／札 | コンテキストに載る要素 | システムプロンプト・スキル・ツール定義・資料など、文脈に置かれるテキスト片。実行中は不変。 |
| デッキ | エージェント構成（の永続部分） | システムプロンプト＋スキル＋ツール＋データ参照＋方針など、対戦をまたいで残る設定一式。 |
| トークン（盤面の生成物） | ―（ゲーム用語）／※下注 | 対戦中に生成され終了で消える派生物（生成テキスト・思考・ツール結果）。 |
| トークン（最小単位） | トークン（AI用語） | テキストを分割した処理の最小単位（サブワード）。課金・長さの単位。 |
| マナ | 計算コスト／トークン課金 | 1回の生成にかかる計算量と料金。入力＋出力トークン量に概ね比例。 |
| ターン | 1回の生成サイクル（1往復） | 入力→生成→応答の1単位。生成と生成の間にモデルは「動いて」いない。 |
| 対戦 | セッション／会話／タスク実行 | 1つの文脈が始まり〜終わる範囲。コンテキストの寿命。 |
| シーズン | モデルの世代（バージョン） | 学習・出荷された別個のモデル。世代交代＝成長でなく別物への置換。 |
| 工場 | 訓練（事前学習・ポストトレーニング） | 大量データから重みを学習する工程。卓上（推論）ではなくここで学ぶ。 |
| 手札 | 未ロードのスキル（名前＋説明のみ） | 必要時に全文が文脈へ展開される、段階的開示（progressive disclosure）の対象。 |
| 墓地 | 過去ログ／会話履歴 | 文脈から外れた過去の生成物。検索すれば呼び戻せる。 |
| 場外の倉庫 | 外部データソース（RAG・外部システム） | 社内文書・Web・DBなど卓外の情報。ツールや検索で取りに行く。 |
| クエスト掲示板 | タスクリスト／TODO・計画 | 達成条件と進捗を保持する仕組み。完了判定の根拠。 |
| 取りに行く札 | ツール／関数呼び出し・MCP | モデルが外部を読む/操作するための実行可能なインターフェース。 |
| コンボ | エージェントループ（tool-useの反復） | 生成→ツール実行→結果を文脈へ→再生成、の繰り返し。 |
| 召喚の入れ子 | サブエージェント／マルチエージェント | 別の文脈で子タスクを実行し、結果だけ親に返す構成。 |
| カードを刷る打ち手 | 自己改善／スキル生成・メモリ書込み | 実行中に手順や学びを永続データ（デッキ）へ書き出すこと。 |
| ドリーミング | メモリの固定化（記憶抽出・永続化） | 対戦後に重要情報を選んで永続メモリへ書き写す処理。 |
| リーグ運営 | オーケストレーション／運用自動化 | タスクの発見・割当・採点・次タスク編成までを仕組み化すること。 |

> 下注：ゲームの「トークン」とAIの「トークン」は**別概念**。
> ただし"対戦中に生まれ、終わると消える"性質を説明するのに偶然うまく重なる。
> 本書ではこの一致を活かしつつ、別物である旨を明記する（誠実さの担保）。

### S. 安全・攻撃：盤面に載る札は信頼できない（比喩が危険を診断する好例）
> 「カードは不変」は作者の札の話。盤面に載る内容が信頼できる保証はない＝AI固有の弱点。
> ただし弱点も対処もカードゲームの語彙の内側で説明できる＝良い比喩の証拠。
- S1 物理カードとの決定的な違い：枠（信頼境界）が見えない／中身しか読まれない
- S2 偽の札が盤面に滑り込む＝プロンプトインジェクションの正体
- S3 外部由来のテキスト（ツール結果・取得文書・Web）が指示になる経路
- S4 同型側の対処：lock凍結＋digest照合／デッキチェック（稼働中監査）
- S5 統制は中身でなく外付け（スーパーセット規律）＝信頼境界を外で強制する
- S6 v2公理A6（敵対者の存在）と、TCGの不正対策30年の移植
- S7 「ジャッジを呼ぶ」＝人間レビュー／承認フローへの接続

### R. 誤りを正す章：競合する比喩の限界と、なぜカードゲームか（★説得の中核）

> 「AIエージェント＝人」「＝ITシステム」も世間でよく使われる比喩。どちらも一部は当たるが
> 破綻する。その破綻を見せた上で、**全部を同時に説明できるのはカードゲームだけ**だと示す。
> ＝読者が既に持っている誤った地図を、正しい地図に置き換える章。

#### R-a 比喩①「AIエージェント＝人」がなぜ違うか
- R1 「優秀な人を採用して育てる」発想の誤り（人は育つ／AIカードは育たない）
- R2 教育・OJT・成長＝デッキ改善であって、打ち手は1ミリも変わらない
- R3 「覚えておいて」「考えておいて」が通じない（人にはある記憶・自発性がAIに無い）
- R4 信頼・評価・モチベ・忠誠といった人の概念が存在しない（札に感情はない）
- R5 1人を育てるより1000卓に複製、という人事ではありえない経済（複製可能性）
- R6 ※ただし"協働の手触り"を掴むには人の比喩も部分的に有効、と公平に注記

#### R-b 比喩②「AIエージェント＝ITシステム／ソフトウェア」がなぜ部分的か
- R7 当たる面：ツール連携・自動化・スケール・APIで外部接続＝システム的に正しい
- R8 限界①決定性：従来システムは決定的・再現的、AIは確率的（同じ入力で違う出力）
- R9 限界②状態：従来は状態を持ち続けられる、AIは対戦ごとにリセット（盤面消滅）
- R10 限界③仕様：仕様書どおりに動く≠プロンプトどおりに動く（保証でなく傾向）
- R11 「バグを直す」ではなく「デッキを組み替える」――修正の作法が違う
- R12 ※自動化・連携の"設計"を考えるにはシステムの比喩は今も有効、と公平に注記

#### R-c なぜカードゲームが一番しっくりくるか
- R13 3比喩の比較表（人／ITシステム／カードゲーム × 各特性で○△×）
- R14 カードゲームだけが同時に説明できる束：
      不変な札（モデル固定）／対戦ごと状態リセット（盤面消滅）／コスト＝マナ／
      構築で価値が貯まる（デッキ）／確率的だがルールはある／世代交代（環境変化）
- R15 「どの比喩も完璧ではない」誠実な結語：
      人＝協働の感覚、システム＝連携の設計、**カード＝本質構造**、と役割分担

#### 比較表ドラフト（収束時に磨く）
| 特性 | 人として見る | ITシステムとして見る | カードゲームとして見る |
